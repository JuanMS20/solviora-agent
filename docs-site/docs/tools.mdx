# Tools

Tools are what turn a chat model into an autonomous agent. Solviora ships a curated toolbox — each tool handles a specific capability like running code, searching the web, or managing files.

## How tools work

When you ask Solviora a question that requires external action (e.g., "read this file" or "search for X"), the agent:

1. Decides which tool to use
2. Calls it with the right arguments
3. Feeds the result back into the conversation
4. Continues reasoning

This happens automatically — you just ask naturally.

## Tool discovery

Tools are auto-discovered at import time. Every Python file in the Solviora tools directory that calls `registry.register()` is loaded automatically — no manual import list to maintain. Each tool returns a JSON string result that feeds back into the agent's reasoning loop.

The agent decides which tool to use, calls it with the right arguments, feeds the result back into the conversation, and continues reasoning. This happens automatically — you just ask naturally.

Tools are organized into **toolsets** — logical groups you can enable or disable per platform:

| Toolset | Tools | Description |

| `messaging` | send_message | Cross-platform message sending |
| `vision` | vision_analyze | Image analysis |
| `image_gen` | image_generate | Image generation |
| `tts` | text_to_speech | Text-to-speech (Edge, ElevenLabs, OpenAI) |

## Enable or disable toolsets

### In config.yaml

```yaml
enabled_toolsets:
  - file
  - web
  - code_execution

disabled_toolsets:
  - image_gen
```

### At runtime

```bash
solviora --enabled-toolsets file,web
solviora --disabled-toolsets image_gen
```

## Terminal backends

Commands can execute on different backends based on your security needs:

| Backend | Isolation | Approval checks | Use case |
|---------|-----------|-----------------|----------|
| `local` | None — runs on host | ✅ Yes | Development |
| `docker` | Container (cap-drop, no-new-privs) | ❌ Skipped (container is boundary) | Production gateway |
| `ssh` | Remote machine | ✅ Yes | Remote server |
| `modal` | Cloud sandbox | ❌ Skipped | Ephemeral cloud compute |
| `daytona` | Cloud container | ❌ Skipped | Persistent workspaces |
| `singularity` | Container (HPC) | ❌ Skipped | Shared clusters |

## Notable tools

### `execute_code`

Run Python scripts that can call tools programmatically — collapses multi-step workflows into a single inference call.

```
You › Run a Python script that calculates fibonacci(50)
```

### `web_search`

Search the web using the configured search backend.

```
You › Search for the latest Python 3.13 release notes
```

### `read_file` / `write_file`

Read and write files on your filesystem with safety bounds.

```
You › Read src/main.py and add type hints
```

### `terminal`

Persistent terminal session for multi-step workflows.

```
You › Create a new git branch, make changes, and run the tests
```

### `memory`

Store and retrieve long-term memories across sessions.

```
You › Remember that I prefer dark mode in all my projects
```

### `delegate_task`

Spawn an isolated subagent with its own conversation and tools for complex subtasks.

### `cronjob`

Schedule recurring tasks in natural language. Runs unattended through the gateway.

## Safety

- Tools operate within your configured working directory by default
- File operations are restricted to allowed paths
- Shell commands are checked against an approval system for dangerous patterns
- Container backends skip local approval checks (the container is the boundary)
- Use `--no-tools` to run without any tools
